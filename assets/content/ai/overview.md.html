生成式 AI (GenAI) 结合大型语言模型 (LLMs) 可用于创建复杂且引人入胜的应用体验，包括个性化内容、智能推荐、媒体生成和理解、信息摘要以及动态功能。

开发此类功能以前需要深厚的领域专业知识和大量的工程投入。然而，新产品和 SDK 正在降低入门门槛。Angular 非常适合将 AI 集成到您的 Web 应用中，原因如下：

  • Angular 强大的模板 API 支持创建由生成内容组成的动态、简洁的 UI
  • 强大的、基于信号的架构，旨在动态管理数据和状态
  • Angular 与 AI SDK 和 API 无缝集成

本指南演示了如何使用 Genkit、Firebase AI Logic 和 Gemini API 为您的 Angular 应用注入 AI。本指南将通过解释如何开始将 AI 集成到 Angular 应用中，来启动您的 AI 驱动的 Web 应用开发之旅。本指南还分享了入门资源，例如入门套件、示例代码和常见工作流的配方，您可以快速上手。

开始之前，您应该对 Angular 有基本的了解。刚接触 Angular？请尝试我们的基础指南或我们的入门教程。

注意：虽然此页面展示了与 Google AI 产品的集成和示例，但像 Genkit 这样的工具是模型无关的，允许您选择自己的模型。在许多情况下，示例和代码样本适用于其他第三方解决方案。

入门

构建 AI 驱动的应用是一个新兴且快速发展的领域。决定从哪里开始以及选择哪些技术可能具有挑战性。以下部分提供了三种选择：

  1. Genkit 为您提供了用于构建全栈应用的支持的模型和统一 API 接口。非常适合需要复杂后端 AI 逻辑的应用，例如个性化推荐。

  2. Firebase AI Logic 为 Google 的模型提供安全的客户端 API，用于构建仅客户端应用或移动应用。最适合浏览器中的交互式 AI 功能，例如实时文本分析或基本聊天机器人。

  3. Gemini API 使您能够直接构建使用通过 API 表面公开的方法和功能的应用程序，最适合全栈应用。适用于需要直接控制 AI 模型（如自定义图像生成或深度数据处理）的应用。

最佳实践

连接模型提供商并确保 API 凭证安全

连接模型提供商时，务必妥善保管您的 API 密钥。切勿将 API 密钥放入发送到客户端的文件中，例如 environments.ts。

您的应用架构决定了选择哪些 AI API 和工具。具体来说，根据您的应用是客户端还是服务器端来选择。像 Firebase AI Logic 这样的工具为客户端代码提供了与模型 API 的安全连接。如果您想使用 Firebase AI Logic 以外的其他 API，或者更喜欢使用其他模型提供商，请考虑创建一个代理服务器，甚至使用 Cloud Functions for Firebase 作为代理，以避免暴露您的 API 密钥。

有关使用客户端应用进行连接的示例，请参阅代码：Firebase AI Logic Angular 示例存储库。

对于需要 API 密钥的服务器端连接到模型 API，请优先使用密钥管理器或环境变量，而不是 environments.ts。您应该遵循保护 API 密钥和凭证的标准最佳实践。Firebase 现在通过 Firebase App Hosting 的最新更新提供了一个新的密钥管理器。要了解更多信息，请查看官方文档。

使用工具调用增强应用

如果您想构建 Agentic 工作流，即代理能够根据提示采取行动并使用工具来解决问题，请使用“工具调用”。工具调用，也称为函数调用，是一种让 LLM 能够向调用它的应用程序发出请求的方式。作为开发人员，您可以定义哪些工具可用，并控制如何或何时调用这些工具。

工具调用通过将您的 AI 集成扩展到问答式聊天机器人之外，进一步增强了您的 Web 应用。事实上，您可以通过模型提供商的函数调用 API，让您的模型能够请求函数调用。可用的工具可用于在您的应用程序上下文中执行更复杂的操作。

在 Angular 示例存储库的电子商务示例中，LLM 请求调用库存函数以获得执行更复杂任务所需的上下文，例如计算商店中一组商品的价格。可用的 API 范围由您作为开发人员决定，就像是否调用 LLM 请求的函数一样。您仍然可以控制执行流程。例如，您可以公开服务中的特定函数，但不是该服务的所有函数。

处理非确定性响应

由于模型可能返回非确定性结果，因此您的应用程序应考虑到这一点。以下是您可以在应用程序实现中使用的一些策略：

  • 调整提示和模型参数（例如温度）以获得更多或更少的确定性响应。您可以在 ai.google.dev 的提示策略部分找到更多信息。
  • 使用“人工干预”策略，即由人工在工作流中进行下一步操作之前验证输出。构建您的应用程序工作流，允许操作员（人类或其他模型）验证输出并确认关键决策。
  • 利用工具（或函数）调用和模式约束来指导和限制模型响应以符合预定义格式，从而提高响应的可预测性。

即使考虑了这些策略和技术，也应在应用程序设计中加入合理的后备方案。遵循现有的应用程序弹性标准。例如，如果资源或 API 不可用，应用程序崩溃是不可接受的。在这种情况下，会向用户显示错误消息，并在适用时显示后续步骤的选项。构建 AI 驱动的应用程序需要同样的考虑。确认响应与预期输出一致，并在不一致时通过优雅降级提供“安全着陆”。这也适用于 LLM 提供商的 API 故障。

考虑以下示例：LLM 提供商没有响应。处理中断的一种潜在策略是：

  • 保存用户的响应，以便在重试场景中使用（现在或稍后）
  • 向用户发出中断警报，并附带一条不泄露敏感信息的适当消息
  • 在服务恢复可用后，稍后恢复对话。
