注册回调函数，以便在指定的阶段内，每次应用程序完成渲染时调用。可用的阶段有：

  • earlyRead 在此阶段，在后续的 write 回调之前从 DOM 中读取数据，例如执行浏览器本身不支持的自定义布局。如果读取操作可以等到 write 阶段之后再进行，请优先使用 read 阶段。切勿在此阶段向 DOM 中写入数据。
  • write 在此阶段向 DOM 中写入数据。切勿在此阶段从 DOM 中读取数据。
  • mixedReadWrite 在此阶段可以同时从 DOM 中读取和写入数据。如果可能的话，请将工作分配给其他阶段，切勿使用此阶段。
  • read 在此阶段从 DOM 中读取数据。切勿在此阶段向 DOM 中写入数据。

如果可能，您应该优先使用 read 和 write 阶段，而不是 earlyRead 和 mixedReadWrite 阶段，以避免性能下降。

请注意：

  • 回调函数在每次渲染后按以下阶段顺序运行：
    1. earlyRead
    2. write
    3. mixedReadWrite
    4. read
  • 同一阶段的回调函数按注册顺序运行。
  • 回调函数仅在浏览器平台上运行，不会在服务器上运行。

作为此规范一部分运行的第一个阶段回调函数将不接收任何参数。此规范中每个后续阶段的回调函数将接收先前运行的阶段回调函数的返回值作为参数。这可用于协调多个阶段的工作。

Angular 无法验证或强制执行阶段的正确使用，而是依赖每位开发者遵循每个值的文档指南，并仔细选择合适的阶段，必要时重构代码。通过这样做，Angular 能够更好地最小化与手动 DOM 访问相关的性能下降，确保您的应用程序或库的最终用户获得最佳体验。

组件不能保证在回调函数运行前被水合。您在直接读取或写入 DOM 和布局时必须谨慎。

注册一个回调函数，以便在 mixedReadWrite 阶段，每次应用程序完成渲染时调用。

您应该优先为回调函数指定一个明确的阶段，否则可能会导致严重的性能下降。

请注意，回调函数将运行

  • 按注册顺序
  • 每次渲染一次
  • 仅在浏览器平台上
  • 在 mixedReadWrite 阶段期间

组件不能保证在回调函数运行前被水合。您在直接读取或写入 DOM 和布局时必须谨慎。
