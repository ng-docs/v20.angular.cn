注册回调函数，使其在应用下次完成渲染时，在指定的阶段期间被调用。可用的阶段有：

  • earlyRead 使用此阶段在后续的 write 回调之前从 DOM 中读取，例如执行浏览器本身不支持的自定义布局。如果读取可以等到写入阶段之后，请优先使用 read 阶段。绝不在此阶段写入 DOM。
  • write 使用此阶段写入 DOM。绝不在此阶段从 DOM 中读取。
  • mixedReadWrite 使用此阶段同时读取和写入 DOM。如果可以将工作分配到其他阶段，绝不要使用此阶段。
  • read 使用此阶段从 DOM 中读取。绝不在此阶段写入 DOM。

如果可能，您应该优先使用 read 和 write 阶段，而不是 earlyRead 和 mixedReadWrite 阶段，以避免性能下降。

请注意：

  • 回调函数以下列阶段顺序在下次渲染之后运行一次：
    1. earlyRead
    2. write
    3. mixedReadWrite
    4. read
  • 同一阶段中的回调函数按照它们注册的顺序运行。
  • 回调函数仅在浏览器平台上运行，它们不会在服务器上运行。

作为此规范一部分运行的第一个阶段回调将不接收任何参数。此规范中每个后续阶段的回调都将接收先前运行的阶段回调的返回值作为参数。这可以用于协调跨多个阶段的工作。

Angular 无法验证或强制阶段是否被正确使用，而是依赖于每位开发者遵循为每个值记录的指南，并仔细选择合适的阶段，并在必要时重构他们的代码。通过这样做，Angular 能够更好地最大限度地减少与手动 DOM 访问相关的性能下降，从而确保为你的应用或库的最终用户提供最佳体验。

不能保证组件在回调运行之前完成水合。 直接读取或写入 DOM 和布局时，您必须谨慎。

注册一个回调函数，使其在应用下次完成渲染时，在 mixedReadWrite 阶段期间被调用。

您应该优先为回调指定一个显式的阶段，否则您将面临严重的性能下降风险。

请注意，回调函数将运行

  • 按照其注册的顺序
  • 仅在浏览器平台上
  • 在 mixedReadWrite 阶段期间

不能保证组件在回调运行之前完成水合。 直接读取或写入 DOM 和布局时，您必须谨慎。
