注册一个副作用，当它被触发时，将在应用完成渲染期间的 mixedReadWrite 阶段被调用。

你应该优先为副作用指定一个明确的阶段，否则你将面临严重的性能下降风险。

请注意，基于回调的 afterRenderEffect 将运行

  • 在它们被注册的顺序中
  • 仅当发生变化时
  • 仅在浏览器平台
  • 在 mixedReadWrite 阶段期间

无法保证组件在回调运行之前完成水合。在直接读取或写入 DOM 和布局时，你必须谨慎。

注册一些副作用，当它们被触发时，将在应用完成渲染期间的指定阶段被调用。可用的阶段有：

  • earlyRead 使用此阶段在后续的 write 回调之前从 DOM 中读取，例如执行浏览器本身不支持的自定义布局。如果读取可以等到写入阶段之后，则首选 read 阶段。绝不在此阶段写入 DOM。
  • write 使用此阶段写入 DOM。绝不在此阶段从 DOM 中读取。
  • mixedReadWrite 使用此阶段同时读取和写入 DOM。如果可以将工作分配到其他阶段，绝不要使用此阶段。
  • read 使用此阶段从 DOM 中读取。绝不在此阶段写入 DOM。

如果可能，你应该优先使用 read 和 write 阶段，而不是 earlyRead 和 mixedReadWrite 阶段，以避免性能下降。

请注意：

  • 副作用按照以下阶段顺序运行，仅当通过信号依赖项发生变化时：
    1. earlyRead
    2. write
    3. mixedReadWrite
    4. read
  • afterRenderEffect 在同一阶段按注册顺序运行。
  • afterRenderEffect 仅在浏览器平台上运行，不会在服务器上运行。
  • afterRenderEffect 将至少运行一次。

作为此规范一部分运行的第一个阶段回调将不接收任何参数。此规范中的每个后续阶段回调将接收先前运行的阶段回调的返回值作为 Signal。这可用于协调多个阶段的工作。

Angular 无法验证或强制阶段是否被正确使用，而是依赖于每位开发人员遵循为每个值记录的指南，并仔细选择合适的阶段，并在必要时重构其代码。通过这样做，Angular 能够更好地最大限度地减少与手动 DOM 访问相关的性能下降，从而确保你的应用或库的最终用户获得最佳体验。
