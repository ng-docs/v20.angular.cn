Angular 路线图

edit

了解 Angular 团队如何在 Web 上构建势头。

作为一个开源项目，Angular 的每日提交、PR 和发展势头都可在 GitHub 上追踪。为了提高这种日常工作与框架未来之间联系的透明度，我们的路线图汇集了团队当前和未来计划的愿景。

以下项目与特定的 Angular 版本无关。我们将在完成后发布它们，并且根据我们的发布计划，遵循语义化版本控制，它们将成为特定版本的一部分。例如，如果特性包含重大更改，我们会在完成后的下一个小版本或下一个主版本中发布。

目前，Angular 框架有两个目标：

  1. 改进 Angular 开发者体验 以及
  2. 改进框架的性能。

继续阅读以了解我们计划如何通过具体的项目工作来实现这些目标。

改进 Angular 开发者体验

开发者效率

Selectorless

为了减少样板代码并提高独立组件的工效学，我们现在正在设计一种解决方案，该方案将使选择器成为可选的。要使用组件或指令，你将能够导入它并直接在组件的模板中使用它。

我们已经开始了无选择器的早期原型设计，并且仍处于规划的早期阶段。当我们有了设计并准备好进行下一步时，我们将分享一个征求意见稿。

Signal Forms

我们计划分析关于 Angular 表单的现有反馈，并设计一个解决方案，以满足开发人员的需求，并使用 Signal 来管理响应式状态。

异步响应式

为了让开发者能够使用信号处理异步数据流，我们开发了 resource 异步原始类型。在此基础上，我们引入了 httpResource，它允许您发送 HTTP 请求并将响应作为信号接收。

我们仍在积极收集这些新的实验性 API 的反馈。请尝试使用它们，并在 GitHub 上与我们分享您的反馈！

Zoneless Angular

在 v18 中，我们在 Angular 中发布了实验性的 zoneless 支持。它使开发者无需在其捆绑包中包含 zone.js 即可使用该框架，从而提高了性能、调试体验和互操作性。作为初始版本的一部分，我们还在 Angular CDK 和 Angular Material 中引入了 zoneless 支持。

在 v19 中，我们在服务器端渲染中引入了无区域支持，处理了一些边缘情况，并创建了一个原理图来脚手架无区域项目。我们将 Google Fonts 迁移到了无区域，这提高了性能和开发者体验，并使我们能够识别在将此功能移至开发者预览版之前需要解决的差距。

截至 Angular v20.2，Zoneless Angular 已稳定，并包含错误处理和服务器端渲染方面的改进。

Signal 集成

我们正在努力改进诸如表单、HTTP 和路由器等基本 Angular 包与 Signals 的集成。作为此项目的一部分，我们将寻求机会引入便捷的基于 signal 的 API 或包装器，以改善整体开发者体验。

改进 HMR（热模块重载）

我们正在通过启用热模块替换来努力实现更快的编辑/刷新周期。

在 Angular v19 中，我们发布了对 CSS 和模板 HMR 的初始支持，并在 v20 中将模板 HMR 升级为稳定版。我们将继续收集反馈，以确保在将此项目标记为完成后，我们已满足开发者的需求。

增量采用

Angular 一直缺乏工具和灵活性，无法向多页应用添加交互性，或者将 Angular 组件嵌入到使用不同框架构建的现有应用中。

作为本项目的一部分，我们将探索跨框架互操作的需求空间以及我们的构建工具产品，以使这种用例成为可能。

Angular 库使用 Ivy

在 2020 年初，我们分享了一个关于 Ivy 库分发的 RFC。 在收到来自社区的宝贵反馈后，我们制定了该项目的设计方案。 我们现在正在投资开发 Ivy 库分发，包括更新库包格式以使用 Ivy 编译，解除对 View Engine 库格式和 ngcc 弃用的限制。

已于 2021 年第三季度完成

通过自动测试环境清理来缩短测试时间和改进调试

为了缩短测试时间并在各个测试之间创建更好的隔离，我们希望更改 TestBed 以在每次测试运行后自动清理和拆除测试环境。

已于 2021 年第三季度完成

弃用并移除 IE11 支持

Internet Explorer 11 (IE11) 一直在阻止 Angular 利用 Web 平台的一些现代特性。 作为此项目的一部分，我们将弃用并移除 IE11 支持，从而为常青浏览器提供的现代特性开辟道路。 我们发起了一个 RFC，以收集社区的反馈并决定后续步骤。

已于 2021 年第三季度完成

利用 ES2017+ 作为默认输出语言

支持现代浏览器使我们能够利用 JavaScript 更紧凑、更具表现力且性能更高的新语法。 作为此项目的一部分，我们将调查阻碍我们推进这项工作的因素，并采取措施使其成为可能。

已于 2021 年第三季度完成

通过 Angular DevTools 加速调试和性能分析

我们正在开发 Angular 的开发工具，该工具提供用于调试和性能分析的实用程序。 该项目旨在帮助开发人员了解 Angular 应用中的组件结构和变更检测。

已于 2021 年第二季度完成

通过整合 Angular 版本控制和分支来简化发布

我们希望整合 Angular（angular/angular、angular/angular-cli 和 angular/components）的多个 GitHub 仓库之间的发布管理工具。 这项工作使我们能够复用基础设施、统一和简化流程，并提高发布过程的可靠性。

已于 2021 年第二季度完成

通过提交消息标准化提高开发人员的一致性

我们希望统一 Angular 仓库（angular/angular、angular/components 和 angular/angular-cli）之间的提交消息要求和一致性，从而使我们的开发过程保持一致性并复用基础设施工具。

已于 2021 年第二季度完成

将 Angular 语言服务过渡到 Ivy

该项目的目标是通过将语言服务过渡到 Ivy 来改善体验并消除遗留依赖。 今天，即使对于 Ivy 应用，语言服务仍然使用 View Engine 编译器和类型检查。 我们希望为 Angular 语言服务使用 Ivy 模板解析器和改进的类型检查，以匹配应用行为。 这种迁移也是朝着解除对移除 View Engine 的限制迈出的一步，这将简化 Angular，减小 npm 包大小，并提高框架的可维护性。

已于 2021 年第二季度完成

通过 Angular 中的原生 Trusted Types 提高安全性

通过与 Google 安全团队合作，我们正在添加对新的 Trusted Types API 的支持。 这个 Web 平台 API 帮助开发人员构建更安全的 Web 应用。

已于 2021 年第二季度完成

通过 Angular CLI webpack 5 优化构建速度和捆绑包大小

作为 v11 版本发布的一部分，我们在 Angular CLI 中引入了 webpack 5 的选择性预览。 为了确保稳定性，我们将继续迭代该实现，以实现构建速度和捆绑包大小的改进。

已于 2021 年第二季度完成

通过在 Universal 应用中内联关键样式来加快应用速度

加载外部样式表是一个阻塞操作，这意味着浏览器在加载所有引用的 CSS 之前无法开始渲染你的应用。 在页面的标头中包含渲染阻塞资源会严重影响其加载性能，例如，首次内容绘制。 为了加快应用速度，我们一直在与 Google Chrome 团队合作，研究内联关键 CSS 和异步加载其余样式。

已于 2021 年第一季度完成

通过更好的 Angular 错误消息改进调试

错误消息通常带来的可操作信息有限，无法帮助开发人员求解问题。 我们一直在努力通过添加关联代码、开发指南和其他材料来使错误消息更易于发现，以确保更顺畅的调试体验。

已于 2021 年第一季度完成

通过更新的入门文档改进开发者入职体验

我们将重新定义用户学习历程并更新入门文档。 我们将明确说明 Angular 的优势、如何探索其功能，并提供指导，以便开发人员可以在尽可能短的时间内熟练掌握该框架。

已于 2021 年第一季度完成

扩展组件测试工具最佳实践

Angular CDK 在版本 9 中向 Angular 引入了组件测试工具的概念。 测试工具使组件作者可以创建受支持的 API，用于测试组件交互。 我们正在继续改进此工具基础设施，并阐明围绕使用工具的最佳实践。 我们还在努力推动 Google 内部更多地采用工具。

已于 2021 年第一季度完成

编写内容投影指南

内容投影是 Angular 的核心概念，但在文档中没有得到应有的重视。 作为此项目的一部分，我们希望确定内容投影的核心用例和概念，并记录它们。

已于 2021 年第二季度完成

迁移到 ESLint

随着 TSLint 的弃用，我们将迁移到 ESLint。 作为该过程的一部分，我们将努力确保与当前推荐的 TSLint 配置向后兼容，为现有的 Angular 应用实施迁移策略，并在 Angular CLI 工具链中引入新的工具。

已于 2020 年第四季度完成

再见积压操作（也称为 Byelog 操作）

我们正在积极投入高达 50% 的工程能力来分类问题和 PR，直到我们清楚地了解更广泛的社区需求。 在此之后，我们将投入高达 20% 的工程能力来及时跟进新的提交。

已于 2020 年第四季度完成
