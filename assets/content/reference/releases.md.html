Angular 版本控制与发布

edit

我们认识到你需要 Angular 框架的稳定性。稳定性确保了可复用的组件和库、教程、工具以及学到的实践经验不会意外地过时。稳定性对于 Angular 周围生态系统的蓬勃发展至关重要。

我们也和你一样，认为 Angular 需要不断发展。我们努力确保你正在构建的基础设施能够持续改进，使你能够与 Web 生态系统的其他部分以及用户的需求保持同步。

本文档包含我们遵循的实践，旨在为你提供一个领先的应用开发平台，并在稳定性方面取得平衡。我们力求确保未来的变更始终以可预测的方式引入。我们希望每个依赖 Angular 的人都知道何时以及如何添加新特性，并在过时的特性被删除时做好充分准备。

有时，为了创新并与不断发展的最佳实践、不断变化的依赖项或 Web 平台的转变保持同步，破坏性变更（例如删除 API 或特性）是必要的。这些破坏性变更会经历一个弃用过程，这在我们的弃用策略中进行了解释。

为了使这些过渡尽可能简单明了，Angular 团队做出了以下承诺：

  • 我们努力减少破坏性变更的数量，并在可能的情况下提供迁移工具
  • 我们遵循此处描述的弃用策略，以便你有时间将你的应用更新到最新的 API 和最佳实践

有帮助：本文档中描述的实践适用于 Angular 2.0 及更高版本。 如果您当前使用的是 AngularJS，请参阅从 AngularJS 升级。 AngularJS 是 Angular 所有 v1.x 版本的名称。

Angular 版本控制

Angular 版本号指示了发布所引入的更改级别。 这种语义化版本控制的使用有助于您了解更新到新版本可能产生的影响。

Angular 版本号有三个部分：major.minor.patch。例如，版本 7.2.11 表示主版本 7，次版本 2 和补丁级别 11。

版本号会根据版本发布中包含的变更级别进行递增。

变更级别 详情
主版本发布 包含重要的新特性，更新期间预计需要少量但最少的开发者协助。当更新到新的主版本发布时，你可能需要运行更新脚本、重构代码、运行额外的测试以及学习新的 API。
次版本发布 包含新的较小特性。次版本发布是完全向后兼容的；更新期间预计不需要开发者协助，但是你可以选择修改你的应用和库，开始使用版本发布中添加的新 API、特性和功能。我们在次版本中通过扩展支持的版本来更新对等依赖，但是我们不要求项目更新这些依赖。
补丁版本发布 低风险的缺陷修复版本发布。更新期间预计不需要开发者协助。

提示： 从 Angular 版本 7 开始，Angular core 和 CLI 的主版本已对齐。这意味着为了在你开发 Angular 应用时使用 CLI，@angular/core 和 CLI 的版本需要相同。

预览版本发布

我们通过为每个主版本和次版本发布提供 "Next" 和发布候选版本 (rc) 预发布版本，让你预览即将发布的内容：

预发布版本类型 详情
Next 正在积极开发和测试的版本。next 版本由附加了 -next 标识符的版本标记指示，例如 8.1.0-next.0。
发布候选版本 一个特性完整且正在进行最终测试的版本。发布候选版本由附加了 -rc 标识符的版本标记指示，例如版本 8.1.0-rc.0。

最新的 next 或 rc 预发布版本的文档可在 next.angular.dev 上找到。

发布频率

我们致力于定期的版本发布计划，以便你可以计划和协调你的更新，使其与 Angular 的持续发展保持同步。

提示： 日期仅作为一般指导，可能会发生变化。

通常，可以预期以下发布周期：

  • 每 6 个月一次主版本发布
  • 每个主版本发布包含 1-3 个次版本发布
  • 几乎每周都会发布补丁版本和预发布版本（next 或 rc）构建

这种发布节奏使渴望尝鲜的开发者能够在其新特性完全开发并通过我们的代码审查和集成测试流程后立即访问这些特性，同时保持平台的稳定性和可靠性，以满足喜欢在特性经过 Google 和其他使用预发布版本构建的开发者验证后再接收这些特性的生产用户的需求。

LTS 修复

作为一般规则，如果修复是为了解决以下问题之一，则 LTS 版本会考虑该修复：

  • 新发现的安全漏洞，
  • 自 LTS 开始以来，由第三方更改（例如新的浏览器版本）引起的回归。

弃用策略

当 Angular 团队打算移除某个 API 或特性时，它将被标记为 deprecated（已弃用）。当某个 API 过时、被另一个 API 取代或以其他方式停止使用时，就会发生这种情况。已弃用的 API 在其弃用阶段仍然可用，该阶段至少持续两个主要版本（大约一年）。

为了帮助确保您有足够的时间和清晰的更新路径，以下是我们的弃用策略：

弃用阶段 详情
公告 我们会在更新日志中宣布已弃用的 API 和特性。已弃用的 API 会在文档中以删除线形式显示。当我们宣布弃用时，也会同时宣布推荐的更新路径。此外，所有已弃用的 API 在相应的文档中都会用 @deprecated 注解，这使得文本编辑器和 IDE 能够在您的项目依赖它们时提供提示。
弃用周期 当 API 或特性被弃用时，它至少在接下来的两个主要版本（至少 12 个月的时间段）中仍然存在。在此之后，已弃用的 API 和特性将成为移除的候选对象。可以在任何版本中宣布弃用，但已弃用 API 或特性的移除仅在主要版本中发生。在已弃用的 API 或特性被移除之前，它会根据 LTS 支持策略进行维护，这意味着只会修复关键问题和安全问题。
npm 依赖 我们只在主要版本中进行需要更改您的应用的 npm 依赖更新。在次要版本中，我们通过扩展支持的版本来更新对等依赖，但我们不要求项目在未来的主要版本之前更新这些依赖。这意味着在 Angular 的次要版本发布期间，Angular 应用和库中的 npm 依赖更新是可选的。

兼容性策略

Angular 是由许多包、子项目和工具组成的集合。 为防止意外使用私有 API，并让您清楚地了解此处所述实践所涵盖的内容——我们记录了什么是以及什么不是我们的公共 API 表面。 有关详细信息，请参阅 Angular 的受支持公共 API 表面。

为了保证 Angular 的向后兼容性，我们在合并任何更改之前都会运行一系列检查：

  • 单元测试和集成测试
  • 比较更改前后公共 API 表面的类型定义
  • 运行 Google 中所有依赖 Angular 的应用的测试

对公共 API 表面的任何更改均根据先前描述的版本控制、支持和弃用策略进行。在特殊情况下，例如关键安全补丁，修复程序可能会引入向后不兼容的更改。此类特殊情况会附带框架官方沟通渠道上的明确通知。

破坏性变更策略和更新路径

破坏性变更需要您进行一些工作，因为变更后的状态与变更前的状态不向后兼容。您可以在 兼容性策略 中找到此规则的罕见例外情况。破坏性变更的示例包括删除公共 API 或 Angular 类型定义的其他更改、更改调用的时序或更新到 Angular 依赖项的新版本，而新版本本身包含破坏性变更。

为了在 Angular 中出现破坏性变更时为您提供支持：

  • 在移除公共 API 之前，我们遵循 弃用策略
  • 通过 ng update 命令支持更新自动化。它提供了代码转换，我们通常在 Google 的数十万个项目中提前测试过这些转换
  • 在 “Angular 更新指南” 中，提供了如何从一个主要版本更新到另一个主要版本的逐步说明

您可以 ng update 到任何 Angular 版本，前提是满足以下条件：

  • 您要更新到的版本受支持。
  • 您要更新自的版本在您要升级到的版本的一个主要版本之内。

例如，您可以从版本 11 更新到版本 12，前提是版本 12 仍然受支持。 如果您想跨多个主要版本更新，请一次执行一个主要版本的更新。 例如，要从版本 10 更新到版本 12：

  1. 从版本 10 更新到版本 11。
  2. 从版本 11 更新到版本 12。

开发者预览版

有时我们会以“开发者预览版”的标签引入新的 API。这些 API 功能齐全且经过完善，但我们尚未准备好根据我们正常的弃用策略对其进行稳定。

这可能是因为我们希望在稳定之前收集来自真实应用的反馈，或者因为相关的文档或迁移工具尚未完全完成。可以通过 GitHub issue 提供反馈，开发者可以在其中分享他们的经验、报告错误或提出改进建议，以帮助改进该特性。

本文档中描述的策略和实践不适用于标记为“开发者预览版”的 API。此类 API 可能会随时更改，即使在框架的新补丁版本中也是如此。团队应自行决定，使用“开发者预览版”API 的好处是否值得冒语义化版本控制正常使用之外的破坏性变更的风险。

实验性的

这些 API 可能根本不会变得稳定，或者在变得稳定之前会发生重大变化。

本文档中描述的策略和实践不适用于标记为实验性的 API。此类 API 可能会随时更改，即使在框架的新补丁版本中也是如此。团队应自行决定，使用实验性 API 的好处是否值得冒语义化版本控制正常使用之外的破坏性变更的风险。
