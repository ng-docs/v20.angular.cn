使用查询引用组件子级

edit

提示： 本指南假设你已经阅读过《要点指南》。如果你是 Angular 新手，请先阅读它。

组件可以定义查询，以查找子元素并从其注入器中读取值。

开发人员最常使用查询来检索对子组件、指令、DOM 元素等的引用。

所有查询函数都返回反映最新结果的信号。您可以通过调用信号函数来读取结果，包括在响应式上下文（如 computed 和 effect）中。

查询分为两类：视图查询和内容查询。

视图查询

视图查询从组件视图中的元素检索结果 — 即组件自身模板中定义的元素。您可以使用 viewChild 函数查询单个结果。

@Component({  selector: 'custom-card-header',  /*...*/})export class CustomCardHeader {  text: string;}@Component({  selector: 'custom-card',  template: '<custom-card-header>Visit sunny California!</custom-card-header>',})export class CustomCard {  header = viewChild(CustomCardHeader);  headerText = computed(() => this.header()?.text);}

在此示例中，CustomCard 组件查询子项 CustomCardHeader，并在 computed 中使用该结果。

如果查询未找到结果，其值为 undefined。如果目标元素被 @if 隐藏，则可能发生这种情况。Angular 会在您的应用程序状态更改时保持 viewChild 结果的最新。

您还可以使用 viewChildren 函数查询多个结果。

@Component({  selector: 'custom-card-action',  /*...*/})export class CustomCardAction {  text: string;}@Component({  selector: 'custom-card',  template: `    <custom-card-action>Save</custom-card-action>    <custom-card-action>Cancel</custom-card-action>  `,})export class CustomCard {  actions = viewChildren(CustomCardAction);  actionsTexts = computed(() => this.actions().map(action => action.text);}

viewChildren 创建一个包含查询结果数组的信号。

查询永远不会穿透组件边界。 视图查询只能从组件的模板中检索结果。

内容查询

内容查询从组件内容中的元素检索结果 — 即在使用该组件的模板中嵌套在该组件内的元素。您可以使用 contentChild 函数查询单个结果。

@Component({  selector: 'custom-toggle',  /*...*/})export class CustomToggle {  text: string;}@Component({  selector: 'custom-expando',  /*...*/})export class CustomExpando {  toggle = contentChild(CustomToggle);  toggleText = computed(() => this.toggle()?.text);}@Component({   /* ... */  // CustomToggle is used inside CustomExpando as content.    template: `    <custom-expando>      <custom-toggle>Show</custom-toggle>    </custom-expando>  `})export class UserProfile { }

在此示例中，CustomExpando 组件查询子项 CustomToggle，并在 computed 中访问该结果。

如果查询未找到结果，其值为 undefined。如果目标元素不存在或被 @if 隐藏，则可能发生这种情况。Angular 会在您的应用程序状态更改时保持 contentChild 结果的最新。

默认情况下，内容查询仅查找组件的直接子级，而不遍历到后代。

您还可以使用 contentChildren 函数查询多个结果。

@Component({  selector: 'custom-menu-item',  /*...*/})export class CustomMenuItem {  text: string;}@Component({  selector: 'custom-menu',  /*...*/})export class CustomMenu {  items = contentChildren(CustomMenuItem);  itemTexts = computed(() => this.items().map(item => item.text));}@Component({  selector: 'user-profile',  template: `    <custom-menu>      <custom-menu-item>Cheese</custom-menu-item>      <custom-menu-item>Tomato</custom-menu-item>    </custom-menu>  `})export class UserProfile { }

contentChildren 创建一个包含查询结果数组的信号。

查询永远不会穿透组件边界。 内容查询只能从与组件自身相同的模板中检索结果。

必要查询

如果子查询（viewChild 或 contentChild）未找到结果，其值为 undefined。如果目标元素被像 @if 或 @for 这样的控制流语句隐藏，则可能发生这种情况。因此，子查询返回的信号在其值类型中包含 undefined。

在某些情况下，特别是使用 viewChild 时，您确切地知道某个特定的子项始终可用。在其他情况下，您可能希望严格强制要求某个特定的子项存在。对于这些情况，您可以使用必需查询。

@Component({/* ... */})export class CustomCard {  header = viewChild.required(CustomCardHeader);  body = contentChild.required(CustomCardBody);}

如果必要查询未找到匹配的结果，Angular 会报告错误。由于这保证了结果是可用的，因此必要查询不会自动在 Signal 的值类型中包含 undefined。

查询定位器

每个查询装饰器的第一个参数都是其定位器。

在大多数情况下，你都希望使用组件或指令作为定位器。

或者，你可以指定与模板引用变量相对应的字符串定位器。

@Component({  /*...*/  template: `    <button #save>Save</button>    <button #cancel>Cancel</button>  `})export class ActionBar {  saveButton = viewChild<ElementRef<HTMLButtonElement>>('save');}

如果多个元素定义了相同的模板引用变量，则查询会检索第一个匹配的元素。

Angular 不支持将 CSS 选择器用作查询定位器。

查询和注入器树

提示： 有关提供者和 Angular 的注入树的背景知识，请参阅依赖注入。

对于更高级的情况，您可以使用任何 ProviderToken 作为定位符。这允许您根据组件和指令提供者来定位元素。

const SUB_ITEM = new InjectionToken<string>('sub-item');@Component({  /*...*/  providers: [{provide: SUB_ITEM, useValue: 'special-item'}],})export class SpecialItem { }@Component({/*...*/})export class CustomList {  subItemType = contentChild(SUB_ITEM);}

上面的示例使用 InjectionToken 作为定位符，但您可以使用任何 ProviderToken 来定位特定元素。

查询选项

所有查询函数都接受一个选项对象作为第二个参数。这些选项控制查询如何查找其结果。

从元素的注入器中读取特定值

默认情况下，查询定位器指示你要搜索的元素和检索到的值。或者，你可以指定 read 选项以从定位器匹配的元素中检索不同的值。

@Component({/*...*/})export class CustomExpando {  toggle = contentChild(ExpandoContent, {read: TemplateRef});}

上面的示例定位了带有指令 ExpandoContent 的元素，并检索了与该元素关联的 TemplateRef。

开发人员最常使用 read 来检索 ElementRef 和 TemplateRef。

内容后代

默认情况下，contentChildren 查询仅查找组件的直接子项，而不遍历后代。 contentChild 查询默认会遍历后代。

@Component({  selector: 'custom-expando',  /*...*/})export class CustomExpando {  toggle = contentChildren(CustomToggle); // none found  // toggle = contentChild(CustomToggle); // found}@Component({  selector: 'user-profile',  template: `    <custom-expando>      <some-other-component>        <custom-toggle>Show</custom-toggle>      </some-other-component>    </custom-expando>  `})export class UserProfile { }

在上面的示例中，CustomExpando 无法使用 contentChildren 找到 <custom-toggle>，因为它不是 <custom-expando> 的直接子项。通过设置 descendants: true，您可以配置查询以遍历同一模板中的所有后代。但是，查询从不穿透组件以遍历其他模板中的元素。

视图查询没有此选项，因为它们始终遍历到后代。

基于装饰器的查询

提示： 尽管 Angular 团队建议在新项目中使用基于 Signal 的查询函数，但 最初的基于装饰器的查询 API 仍然完全受支持。

或者，你可以通过将相应的装饰器添加到属性来声明查询。基于装饰器的查询的行为方式与基于信号的查询相同，但如下所述的除外。

视图查询

您可以使用 @ViewChild 装饰器查询单个结果。

@Component({  selector: 'custom-card-header',  /*...*/})export class CustomCardHeader {  text: string;}@Component({  selector: 'custom-card',  template: '<custom-card-header>Visit sunny California!</custom-card-header>',})export class CustomCard {  @ViewChild(CustomCardHeader) header: CustomCardHeader;  ngAfterViewInit() {    console.log(this.header.text);  }}

在此范例中，CustomCard 组件查询子级 CustomCardHeader 并在 ngAfterViewInit 中访问该结果。

Angular 会在您的应用程序状态更改时保持 @ViewChild 的结果最新。

视图查询结果在 ngAfterViewInit 生命周期方法中变为可用。在此之前，该值为 undefined。有关组件生命周期的详细信息，请参阅生命周期部分。

您也可以使用 @ViewChildren 装饰器查询多个结果。

@Component({  selector: 'custom-card-action',  /*...*/})export class CustomCardAction {  text: string;}@Component({  selector: 'custom-card',  template: `    <custom-card-action>Save</custom-card-action>    <custom-card-action>Cancel</custom-card-action>  `,})export class CustomCard {  @ViewChildren(CustomCardAction) actions: QueryList<CustomCardAction>;  ngAfterViewInit() {    this.actions.forEach(action => {      console.log(action.text);    });  }}

@ViewChildren 创建一个包含查询结果的 QueryList 对象。您可以通过 changes 属性订阅查询结果随时间的更改。

内容查询

您可以使用 @ContentChild 装饰器查询单个结果。

@Component({  selector: 'custom-toggle',  /*...*/})export class CustomToggle {  text: string;}@Component({  selector: 'custom-expando',  /*...*/})export class CustomExpando {  @ContentChild(CustomToggle) toggle: CustomToggle;  ngAfterContentInit() {    console.log(this.toggle.text);  }}@Component({  selector: 'user-profile',  template: `    <custom-expando>      <custom-toggle>Show</custom-toggle>    </custom-expando>  `})export class UserProfile { }

在本范例中，CustomExpando 组件查询子组件 CustomToggle，并在 ngAfterContentInit 中访问查询结果。

Angular 会在您的应用程序状态更改时保持 @ContentChild 的结果最新。

内容查询结果在生命周期方法 ngAfterContentInit 中变为可用。在此之前，该值为 undefined。关于组件生命周期的更多详情，请参阅 生命周期 部分。

您也可以使用 @ContentChildren 装饰器查询多个结果。

@Component({  selector: 'custom-menu-item',  /*...*/})export class CustomMenuItem {  text: string;}@Component({  selector: 'custom-menu',  /*...*/})export class CustomMenu {  @ContentChildren(CustomMenuItem) items: QueryList<CustomMenuItem>;  ngAfterContentInit() {    this.items.forEach(item => {      console.log(item.text);    });  }}@Component({  selector: 'user-profile',  template: `    <custom-menu>      <custom-menu-item>Cheese</custom-menu-item>      <custom-menu-item>Tomato</custom-menu-item>    </custom-menu>  `})export class UserProfile { }

@ContentChildren 创建一个包含查询结果的 QueryList 对象。您可以通过 changes 属性订阅查询结果随时间的更改。

基于装饰器的查询选项

所有查询装饰器都接受一个选项对象作为第二个参数。这些选项的工作方式与基于信号的查询相同，但下述情况除外。

静态查询

@ViewChild 和 @ContentChild 装饰器接受 static 选项。

@Component({  selector: 'custom-card',  template: '<custom-card-header>Visit sunny California!</custom-card-header>',})export class CustomCard {  @ViewChild(CustomCardHeader, {static: true}) header: CustomCardHeader;  ngOnInit() {    console.log(this.header.text);  }}

通过设置 static: true，你可以向 Angular 保证此查询的目标 始终 存在，并且不是有条件渲染的。这使得结果更早可用，在 ngOnInit 生命周期方法中。

静态查询结果在初始化之后不会更新。

static 选项不适用于 @ViewChildren 和 @ContentChildren 查询。

使用 QueryList

@ViewChildren 和 @ContentChildren 都提供一个 QueryList 对象，其中包含一个结果列表。

QueryList 提供了一些便捷的 API，用于以类似数组的方式处理结果，例如 map、reduce 和 forEach。你可以通过调用 toArray 来获取当前结果的数组。

你可以订阅 changes 属性，以便在结果发生任何更改时执行某些操作。

常见的查询陷阱

使用查询时，常见的陷阱可能会使你的代码更难理解和维护。

始终为多个组件之间共享的状态维护单一事实之源。这避免了不同组件中重复状态变得不同步的情况。

避免直接将状态写入子组件。这种模式可能会导致脆弱的代码，难以理解，并且容易出现 ExpressionChangedAfterItHasBeenChecked 错误。

永远不要直接将状态写入父组件或祖先组件。这种模式可能会导致脆弱的代码，难以理解，并且容易出现 ExpressionChangedAfterItHasBeenChecked 错误。
