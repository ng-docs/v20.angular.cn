Angular 中的渲染策略

edit

本指南可帮助您为 Angular 应用的不同部分选择正确的渲染策略。

什么是渲染策略？

渲染策略决定了您的 Angular 应用的 HTML 内容何时以及在何处生成。每种策略在初始页面加载性能、交互性、SEO 功能和服务器资源使用之间提供了不同的权衡。

Angular 支持三种主要的渲染策略：

  • 客户端渲染 (CSR) - 内容完全在浏览器中渲染
  • 静态站点生成 (SSG/预渲染) - 内容在构建时预先渲染
  • 服务器端渲染 (SSR) - 内容在服务器上为路由的初始请求进行渲染

客户端渲染 (CSR)

CSR 是 Angular 的默认设置。 JavaScript 加载后，内容完全在浏览器中渲染。

何时使用 CSR

✅ 适用于以下场景：

  • 交互式应用（仪表盘、管理面板）
  • 实时应用
  • 不关心 SEO 的内部工具
  • 具有复杂客户端状态的单页应用

❌ 可能时，请考虑避免使用：

  • 需要 SEO 的面向公众的内容
  • 初始加载性能至关重要的页面

CSR 的权衡

方面 影响
SEO 差 - 在 JS 执行之前爬虫无法看到内容
初始加载 较慢 - 必须先下载并执行 JavaScript
交互性 加载后即时可用
服务器需求 除了一些配置外，最少
复杂性 最简单，因为它只需最少的配置即可工作

静态站点生成 (SSG/预渲染)

SSG 在构建时将页面预渲染为静态 HTML 文件。服务器在初始页面加载时发送预构建的 HTML。水合后，您的应用将完全在浏览器中运行，就像传统的 SPA 一样 - 后续的导航、路由更改和 API 调用都发生在客户端，无需服务器渲染。

何时使用 SSG

✅ 适用于以下场景：

  • 营销页面和着陆页
  • 博客文章和文档
  • 内容稳定的产品目录
  • 不因用户而异的内容

❌ 可能时，请考虑避免使用：

  • 用户特定的内容
  • 频繁变化的数据
  • 实时信息

SSG 的权衡

方面 影响
SEO 极佳 - 立即提供完整的 HTML
初始加载 最快 - 预生成的 HTML
交互性 水合完成后
服务器需求 服务时无（CDN 友好）
构建时间 较长 - 预先生成所有页面
内容更新 需要重新构建和重新部署

📖 实现：请参阅 SSR 指南中的自定义构建时预渲染。

服务器端渲染 (SSR)

SSR 为路由的初始请求在服务器上生成 HTML，提供具有良好 SEO 的动态内容。服务器渲染 HTML 并将其发送到客户端。

客户端渲染页面后，Angular 会水合应用，然后应用将完全在浏览器中运行，就像传统的 SPA 一样 - 后续的导航、路由更改和 API 调用都发生在客户端，无需额外的服务器渲染。

何时使用 SSR

✅ 适用于以下场景：

  • 电子商务产品页面（动态定价/库存）
  • 新闻网站和社交媒体信息流
  • 个性化内容，频繁变化

❌ 可能时，请考虑避免使用：

  • 静态内容（请改用 SSG）
  • 当服务器成本是顾虑时

SSR 的权衡

方面 影响
SEO 优秀 - 为爬虫提供完整的 HTML
初始加载 快速 - 内容即时可见
交互性 延迟到水合完成
服务器需求 需要服务器
个性化 完全访问用户上下文
服务器成本 更高 - 在路由的初始请求时渲染

📖 实现：请参阅 SSR 指南中的 服务器路由 和 编写服务器兼容组件。

选择正确的策略

决策矩阵

如果你需要... 使用此策略 原因
SEO + 静态内容 SSG 预渲染的 HTML，最快的加载速度
SEO + 动态内容 SSR 路由初始请求时的最新内容
无 SEO + 交互性 CSR 最简单，无需服务器
混合需求 混合 每个路由使用不同的策略

通过水合使 SSR/SSG 具有交互性

使用 SSR 或 SSG 时，Angular 会“水合”服务器渲染的 HTML 以使其具有交互性。

可用策略：

  • 完全水合 - 整个应用一次性变得可交互（默认）
  • 增量水合 - 部分内容按需变得可交互（性能更佳）
  • 事件重放 - 在水合完成前捕获点击事件
