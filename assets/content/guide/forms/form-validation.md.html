验证表单输入

edit

你可以通过验证用户输入的准确性和完整性来提高总体数据质量。 本页介绍如何在响应式表单和模板驱动表单中，验证来自 UI 的用户输入并显示有用的验证消息。

在模板驱动表单中验证输入

要向模板驱动表单添加验证，你可以添加与 原生 HTML 表单验证 相同的验证 Attribute。 Angular 使用指令将这些 Attribute 与框架中的验证器函数进行匹配。

每次表单控件的值发生变化时，Angular 都会运行验证，并生成验证错误列表（导致 INVALID 状态）或 null（导致 VALID 状态）。

然后，您可以通过将 ngModel 导出到本地模板变量来检查控件的状态。 以下示例将 NgModel 导出到一个名为 name 的变量中：

请注意该范例阐明的以下特性。

  • <input> 元素带有 HTML 验证 Attribute：required 和 minlength。 它还带有一个自定义验证器指令 forbiddenName。 有关更多信息，请参阅自定义验证器部分。

  • #name="ngModel" 将 NgModel 导出到一个名为 name 的本地变量中。 NgModel 镜像了其底层 FormControl 实例的许多属性，因此您可以在模板中使用它来检查控件的状态，例如 valid 和 dirty。 有关控件属性的完整列表，请参阅AbstractControl API 参考。

提示： 为了防止验证器在用户有机会编辑表单之前显示错误，你应该检查控件的 dirty 或 touched 状态。

  • 当用户更改被监视字段中的值时，控件将被标记为 “dirty”
  • 当用户模糊表单控件元素时，控件将被标记为 “touched”

在响应式表单中验证输入

在响应式表单中，事实之源是组件类。 你无需通过模板中的 Attribute 添加验证器，而是将验证器函数直接添加到组件类中的表单控件模型。 然后，每当控件的值发生更改时，Angular 都会调用这些函数。

验证器函数

验证器函数可以是同步的，也可以是异步的。

验证器类型 详情
同步验证器 同步函数，它们接受一个控件实例并立即返回一组验证错误或 null。在实例化 FormControl 时，将它们作为第二个参数传递。
异步验证器 异步函数，它们接受一个控件实例并返回一个 Promise 或 Observable，该 Promise 或 Observable 稍后会发出一个验证错误集或 null。在实例化 FormControl 时，将它们作为第三个参数传递。

出于性能原因，仅当所有同步验证器都通过时，Angular 才会运行异步验证器。 每个验证器都必须在设置错误之前完成。

内置验证器函数

你可以选择编写自己的验证器函数，也可以使用 Angular 的一些内置验证器。

与模板驱动表单中可用的属性（如 required 和 minlength）相同的内置验证器，都可以从 Validators 类作为函数使用。 有关内置验证器的完整列表，请参阅Validators API 参考。

要将演员表单更新为响应式表单，请使用一些相同的内置验证器——这次是以函数形式，如下例所示。

reactive/actor-form-reactive.component.ts (验证器函数)

import {Component} from '@angular/core';import {FormControl, FormGroup, Validators, ReactiveFormsModule} from '@angular/forms';import {forbiddenNameValidator} from '../shared/forbidden-name.directive';@Component({  selector: 'app-actor-form-reactive',  templateUrl: './actor-form-reactive.component.html',  styleUrls: ['./actor-form-reactive.component.css'],  imports: [ReactiveFormsModule],})export class HeroFormReactiveComponent {  skills = ['Method Acting', 'Singing', 'Dancing', 'Swordfighting'];  actor = {name: 'Tom Cruise', role: 'Romeo', skill: this.skills[3]};  actorForm = new FormGroup({    name: new FormControl(this.actor.name, [      Validators.required,      Validators.minLength(4),      forbiddenNameValidator(/bob/i), // <-- Here's how you pass in the custom validator.    ]),    role: new FormControl(this.actor.role),    skill: new FormControl(this.actor.skill, Validators.required),  });  get name() {    return this.actorForm.get('name');  }  get skill() {    return this.actorForm.get('skill');  }}

在此示例中，name 控件设置了两个内置验证器——Validators.required 和 Validators.minLength(4)——以及一个自定义验证器 forbiddenNameValidator。

所有这些验证器都是同步的，因此它们作为第二个参数传递。 请注意，你可以通过将函数作为数组传入来支持多个验证器。

此范例还添加了一些 getter 方法。 在响应式表单中，你始终可以通过父组上的 get 方法访问任何表单控件，但有时定义 getter 作为模板的简写很有用。

如果再次查看 name 输入的模板，它与模板驱动的范例非常相似。

定义自定义验证器

内置验证器并不总是与你的应用的精确用例相匹配，因此有时你需要创建一个自定义验证器。

考虑一下上一个范例中的 forbiddenNameValidator 函数。 以下是该函数的定义。

shared/forbidden-name.directive.ts (forbiddenNameValidator)

import {Directive, forwardRef, input} from '@angular/core';import {  AbstractControl,  NG_VALIDATORS,  ValidationErrors,  Validator,  ValidatorFn,} from '@angular/forms';/** An actor's name can't match the given regular expression */export function forbiddenNameValidator(nameRe: RegExp): ValidatorFn {  return (control: AbstractControl): ValidationErrors | null => {    const forbidden = nameRe.test(control.value);    return forbidden ? {forbiddenName: {value: control.value}} : null;  };}@Directive({  selector: '[appForbiddenName]',  providers: [    {      provide: NG_VALIDATORS,      useExisting: forwardRef(() => ForbiddenValidatorDirective),      multi: true,    },  ],})export class ForbiddenValidatorDirective implements Validator {  readonly forbiddenName = input<string>('', {alias: 'appForbiddenName'});  validate(control: AbstractControl): ValidationErrors | null {    return this.forbiddenName      ? forbiddenNameValidator(new RegExp(this.forbiddenName(), 'i'))(control)      : null;  }}

该函数是一个工厂，它接受一个正则表达式来检测特定的禁用名称，并返回一个验证器函数。

在此示例中，禁用的名称是 “bob”，因此验证器会拒绝任何包含 “bob” 的演员姓名。 在其他地方，它可能会拒绝 “alice” 或配置正则表达式匹配的任何名称。

forbiddenNameValidator 工厂返回配置的验证器函数。 该函数接受一个 Angular 控件对象，如果控件值有效，则返回 null，否则返回一个验证错误对象。 验证错误对象通常具有一个属性（Property），其名称是验证键 'forbiddenName'，其值是可以插入到错误消息中的任意值字典 {name}。

自定义异步验证器与同步验证器类似，但它们必须返回 Promise 或可观察者，后者稍后会发出 null 或验证错误对象。 对于可观察者，可观察者必须完成，此时表单将使用为验证发出的最后一个值。

向模板驱动表单添加自定义验证器

在模板驱动表单中，向模板添加一个指令，其中该指令包装了验证器函数。 例如，相应的 ForbiddenValidatorDirective 充当 forbiddenNameValidator 的包装器。

Angular 识别出指令在验证过程中的作用，因为指令通过 NG_VALIDATORS 提供者进行注册，如下例所示。 NG_VALIDATORS 是一个预定义的提供者，具有可扩展的验证器集合。

shared/forbidden-name.directive.ts (提供者)

import {Directive, forwardRef, input} from '@angular/core';import {  AbstractControl,  NG_VALIDATORS,  ValidationErrors,  Validator,  ValidatorFn,} from '@angular/forms';/** An actor's name can't match the given regular expression */export function forbiddenNameValidator(nameRe: RegExp): ValidatorFn {  return (control: AbstractControl): ValidationErrors | null => {    const forbidden = nameRe.test(control.value);    return forbidden ? {forbiddenName: {value: control.value}} : null;  };}@Directive({  selector: '[appForbiddenName]',  providers: [    {      provide: NG_VALIDATORS,      useExisting: forwardRef(() => ForbiddenValidatorDirective),      multi: true,    },  ],})export class ForbiddenValidatorDirective implements Validator {  readonly forbiddenName = input<string>('', {alias: 'appForbiddenName'});  validate(control: AbstractControl): ValidationErrors | null {    return this.forbiddenName      ? forbiddenNameValidator(new RegExp(this.forbiddenName(), 'i'))(control)      : null;  }}

然后，指令类实现 Validator 接口，以便轻松地与 Angular 表单集成。 这是指令的其余部分，可以帮助您了解它是如何组合在一起的。

跨字段验证

跨字段验证器是一种 自定义验证器，它比较表单中不同字段的值，并接受或拒绝它们的组合。 例如，您可能有一个表单提供相互不兼容的选项，因此用户可以选择 A 或 B，但不能同时选择。 某些字段值也可能依赖于其他字段；用户可能只在选择了 A 的情况下才能选择 B。

以下跨验证范例展示了如何执行以下操作：

  • 根据两个同级控件的值，验证响应式或基于模板的表单输入，
  • 在用户与表单交互且验证失败后，显示描述性错误消息。

这些范例使用跨验证来确保演员在填写“演员表单”时不会在其角色中重复使用相同的名字。 验证器通过检查演员姓名和角色是否不匹配来实现这一点。

向响应式表单添加跨验证

该表单具有以下结构：

const actorForm = new FormGroup({  'name': new FormControl(),  'role': new FormControl(),  'skill': new FormControl()});

请注意，name 和 role 是同级控件。 要在单个自定义验证器中评估这两个控件，您必须在公共祖先控件中执行验证：FormGroup。 您可以查询 FormGroup 的子控件，以便比较它们的值。

要将验证器添加到 FormGroup，请在创建时将新验证器作为第二个参数传递。

const actorForm = new FormGroup({  'name': new FormControl(),  'role': new FormControl(),  'skill': new FormControl()}, { validators: unambiguousRoleValidator });

验证器代码如下所示。

shared/unambiguous-role.directive.ts

import {Directive, forwardRef} from '@angular/core';import {  AbstractControl,  NG_VALIDATORS,  ValidationErrors,  Validator,  ValidatorFn,} from '@angular/forms';/** An actor's name can't match the actor's role */export const unambiguousRoleValidator: ValidatorFn = (  control: AbstractControl,): ValidationErrors | null => {  const name = control.get('name');  const role = control.get('role');  return name && role && name.value === role.value ? {unambiguousRole: true} : null;};@Directive({  selector: '[appUnambiguousRole]',  providers: [    {      provide: NG_VALIDATORS,      useExisting: forwardRef(() => UnambiguousRoleValidatorDirective),      multi: true,    },  ],})export class UnambiguousRoleValidatorDirective implements Validator {  validate(control: AbstractControl): ValidationErrors | null {    return unambiguousRoleValidator(control);  }}

unambiguousRoleValidator 验证器实现了 ValidatorFn 接口。 它接收一个 Angular 控件对象作为参数，并在表单有效时返回 null，否则返回 ValidationErrors。

验证器通过调用 FormGroup 的 get 方法来检索子控件，然后比较 name 和 role 控件的值。

如果值不匹配，则角色是明确的，两者都有效，并且验证器返回 null。 如果它们匹配，则演员的角色是模棱两可的，并且验证器必须通过返回错误对象将表单标记为无效。

创建异步验证器

异步验证器实现了 AsyncValidatorFn 和 AsyncValidator 接口。 这些与它们的同步对应项非常相似，但有以下区别。

  • validate() 函数必须返回 Promise 或可观察者，
  • 返回的可观察者必须是有限的，这意味着它必须在某个时候完成。 要将无限可观察者转换为有限可观察者，请通过诸如 first、last、take 或 takeUntil 之类的过滤操作符管道化该可观察者。

异步验证发生在同步验证之后，并且仅在同步验证成功时执行。 如果更基本的验证方法已经发现无效输入，则此检查使表单可以避免潜在的昂贵异步验证过程（例如 HTTP 请求）。

异步验证开始后，表单控件进入 pending 状态。 检查控件的 pending 属性，并使用它来提供有关正在进行的验证操作的视觉反馈。

常见的 UI 模式是在执行异步验证时显示加载动画。 以下范例展示了如何在模板驱动表单中实现此目的。

<input [(ngModel)]="name" #model="ngModel" appSomeAsyncValidator>@if(model.pending) {  <app-spinner />}

实现自定义异步验证器

在以下范例中，异步验证器确保为演员分配的角色尚未被占用。 新演员不断试镜，老演员不断退休，因此无法提前检索可用角色列表。 为了验证潜在的角色条目，验证器必须启动异步操作以查询当前所有已分配角色的中央数据库。

以下代码创建了验证器类 UniqueRoleValidator，它实现了 AsyncValidator 接口。

import {Directive, forwardRef, inject, Injectable} from '@angular/core';import {  AsyncValidator,  AbstractControl,  NG_ASYNC_VALIDATORS,  ValidationErrors,} from '@angular/forms';import {catchError, map} from 'rxjs/operators';import {ActorsService} from './actors.service';import {Observable, of} from 'rxjs';@Injectable({providedIn: 'root'})export class UniqueRoleValidator implements AsyncValidator {  private readonly actorsService = inject(ActorsService);  validate(control: AbstractControl): Observable<ValidationErrors | null> {    return this.actorsService.isRoleTaken(control.value).pipe(      map((isTaken) => (isTaken ? {uniqueRole: true} : null)),      catchError(() => of(null)),    );  }}@Directive({  selector: '[appUniqueRole]',  providers: [    {      provide: NG_ASYNC_VALIDATORS,      useExisting: forwardRef(() => UniqueRoleValidatorDirective),      multi: true,    },  ],})export class UniqueRoleValidatorDirective implements AsyncValidator {  private readonly validator = inject(UniqueRoleValidator);  validate(control: AbstractControl): Observable<ValidationErrors | null> {    return this.validator.validate(control);  }}

actorsService 属性使用 ActorsService 令牌的实例进行初始化，该令牌定义了以下接口。

interface ActorsService {  isRoleTaken: (role: string) => Observable<boolean>;}

在实际应用中，ActorsService 将负责向演员数据库发出 HTTP 请求，以检查角色是否可用。 从验证器的角度来看，服务的实际实现并不重要，因此该范例可以仅针对 ActorsService 接口进行编码。

当验证开始时，UnambiguousRoleValidator 使用当前控件值委托给 ActorsService 的 isRoleTaken() 方法。 此时，控件被标记为 pending 并保持此状态，直到从 validate() 方法返回的可观察者链完成。

isRoleTaken() 方法调度一个 HTTP 请求，该请求检查角色是否可用，并返回 Observable<boolean> 作为结果。 validate() 方法通过 map 操作符管道化响应，并将其转换为验证结果。

然后，该方法像任何验证器一样，在表单有效时返回 null，在无效时返回 ValidationErrors。 此验证器使用 catchError 操作符处理任何潜在错误。 在这种情况下，验证器将 isRoleTaken() 错误视为成功验证，因为无法发出验证请求并不一定意味着角色无效。 您可以以不同的方式处理错误，而是返回 ValidationError 对象。

一段时间过后，可观察者链完成，异步验证完成。 pending 标志设置为 false，并且表单有效性已更新。

优化异步验证器的性能

默认情况下，所有验证器都会在每次表单值更改后运行。 对于同步验证器，这通常不会对应用性能产生明显影响。 然而，异步验证器通常会执行某种 HTTP 请求来验证控件。 每次击键后都发送 HTTP 请求可能会给后端 API 带来压力，应尽可能避免。

你可以通过将 updateOn 属性从 change （默认值）更改为 submit 或 blur 来延迟更新表单的有效性。

对于模板驱动表单，在模板中设置该属性。

<input [(ngModel)]="name" [ngModelOptions]="{updateOn: 'blur'}">

对于响应式表单，在 FormControl 实例中设置该属性。

new FormControl('', {updateOn: 'blur'});

与原生 HTML 表单验证的交互

默认情况下，Angular 通过在封闭的 <form> 上添加 novalidate 属性来禁用 原生 HTML 表单验证，并使用指令将这些属性与框架中的验证器函数进行匹配。 如果你想结合使用原生验证和基于 Angular 的验证，你可以使用 ngNativeValidate 指令重新启用它。 有关详细信息，请参阅 API 文档。
