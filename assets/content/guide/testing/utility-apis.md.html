测试实用工具 API

edit

本页介绍了最有用的 Angular 测试特性。

Angular 测试工具包括 TestBed、ComponentFixture 以及一些控制测试环境的函数。 TestBed 和 ComponentFixture 类将单独介绍。

下面是独立函数的摘要，按实用程序排序：

函数 详情
waitForAsync 在特殊的异步测试 Zone 中运行测试 (it) 或 setup (beforeEach) 函数的主体。请参阅 waitForAsync。
fakeAsync 在特殊的 fakeAsync 测试 Zone 中运行测试 (it) 的主体，从而实现线性控制流编码风格。请参阅 fakeAsync。
tick 通过刷新 fakeAsync 测试 Zone 中的 timer 和 micro-task 队列，模拟时间的流逝和待处理异步活动的完成。好奇的读者可能会喜欢这篇长篇博文，“Tasks, microtasks, queues and schedules”。接受一个可选参数，该参数使虚拟时钟向前移动指定的毫秒数，从而清除在该时间范围内调度的异步活动。请参阅 tick。
inject 将一个或多个服务从当前的 TestBed 注入器注入到测试函数中。它不能注入由组件本身提供的服务。请参阅关于 debugElement.injector 的讨论。
discardPeriodicTasks 当 fakeAsync() 测试以待处理的定时器事件任务（已排队的 setTimeOut 和 setInterval 回调）结束时，测试将失败并显示清晰的错误消息。
通常情况下，测试应在没有待处理任务的情况下结束。当预期存在待处理定时器任务时，请调用 discardPeriodicTasks 来清空任务队列并避免错误。
flushMicrotasks 当 fakeAsync() 测试以待处理的微任务（例如未解析的 Promise）结束时，测试将失败并显示清晰的错误消息。
通常情况下，测试应等待微任务完成。当预期存在待处理微任务时，请调用 flushMicrotasks 来清空微任务队列并避免错误。
ComponentFixtureAutoDetect 用于开启自动变更检测的服务的提供者令牌。
getTestBed 获取 TestBed 的当前实例。通常是不必要的，因为 TestBed 类的静态类方法通常已足够。 TestBed 实例公开了一些很少使用但无法作为静态方法访问的成员。

TestBed 类摘要

TestBed 类是主要的 Angular 测试工具之一。 它的 API 非常庞大，在您一点一点地探索它之前可能会让人不知所措。 在尝试理解整个 API 之前，请先阅读本指南的开头部分以掌握基础知识。

传递给 configureTestingModule 的模块定义是 @NgModule 元数据属性的一个子集。

type TestModuleMetadata = {  providers?: any[];  declarations?: any[];  imports?: any[];  schemas?: Array<SchemaMetadata | any[]>;};

每个覆盖方法都接受一个 MetadataOverride<T>，其中 T 是适合该方法的元数据类型，即 @NgModule、@Component、@Directive 或 @Pipe 的参数。

type MetadataOverride<T> = {  add?: Partial<T>;  remove?: Partial<T>;  set?: Partial<T>;};

TestBed API 由静态类方法组成，这些方法更新或引用 TestBed 的全局实例。

在内部，所有静态方法都覆盖当前运行时 TestBed 实例的方法，该实例也由 getTestBed() 函数返回。

在 beforeEach() 内部调用 TestBed 方法，以确保在每个单独的测试之前都有一个全新的开始。

以下是最重要的静态方法，按实用程序排序。

方法 详情
configureTestingModule 测试腻子脚本 (karma-test-shim, browser-test-shim) 建立初始测试环境和默认测试模块。默认测试模块配置了基本对象声明和一些每个测试者都需要的 Angular 服务替代品。
调用 configureTestingModule 通过添加和删除导入、对象声明（组件、指令和管道）以及提供者来优化特定测试集的测试模块配置。
compileComponents 在完成配置后异步编译测试模块。如果测试模块的任何组件具有 templateUrl 或 styleUrls，则必须调用此方法，因为获取组件模板和样式文件是异步的。请参阅 compileComponents。
调用 compileComponents 后，TestBed 配置将在当前 spec 的持续时间内被冻结。
createComponent<T> 根据当前的 TestBed 配置创建一个类型为 T 的组件实例。调用 createComponent 后，TestBed 配置将在当前 spec 的持续时间内被冻结。
overrideModule 替换给定 NgModule 的元数据。请注意，模块可以导入其他模块。overrideModule 方法可以深入到当前测试模块以修改这些内部模块之一。
overrideComponent 替换给定组件类的元数据，该组件类可能深深地嵌套在内部模块中。
overrideDirective 替换给定指令类的元数据，该指令类可能深深地嵌套在内部模块中。
overridePipe 替换给定管道类的元数据，该管道类可能深深地嵌套在内部模块中。
inject 从当前的 TestBed 注入器检索服务。inject 函数通常足以满足此目的。但是，如果 inject 无法提供服务，它会抛出错误。
如果服务是可选的怎么办？
TestBed.inject() 方法接受一个可选的第二个参数，即如果 Angular 找不到提供者时要返回的对象（在此示例中为 null）：调用 TestBed.inject 后，TestBed 配置将在当前 spec 的持续时间内被冻结。
initTestEnvironment 为整个测试运行初始化测试环境。
测试垫片（karma-test-shim, browser-test-shim）会为您调用它，因此您很少需要自己调用它。
仅调用此方法一次。要在测试运行期间更改此默认设置，请先调用 resetTestEnvironment。
指定 Angular 编译器工厂、PlatformRef 和默认 Angular 测试模块。非浏览器平台的替代方案可在通用形式 @angular/platform-<platform_name>/testing/<platform_name> 中找到。
resetTestEnvironment 重置初始测试环境，包括默认测试模块。

TestBed 的一些实例方法没有被静态 TestBed 类方法覆盖。 这些方法很少需要。

ComponentFixture

TestBed.createComponent<T> 创建组件 T 的实例，并返回该组件的强类型化 ComponentFixture。

ComponentFixture 的属性和方法提供了对组件、其 DOM 表示以及其 Angular 环境各方面的访问。

ComponentFixture 属性

以下是测试人员最重要的属性，按实用程序排序。

属性 详情
componentInstance 由 TestBed.createComponent 创建的组件类的实例。
debugElement 与组件根元素关联的 DebugElement。
debugElement 在测试和调试期间提供对组件及其 DOM 元素的洞察。它是测试人员的关键属性。最有趣的成员将在 下方介绍。
nativeElement 组件根目录下的原生 DOM 元素。
changeDetectorRef 组件的 ChangeDetectorRef。
当测试具有 ChangeDetectionStrategy.OnPush 方法的组件，或者组件的变更检测在你的程序控制之下时，ChangeDetectorRef 最有价值。

ComponentFixture 方法

fixture 方法使 Angular 对组件树执行某些任务。 调用这些方法以响应模拟的用户操作来触发 Angular 行为。

以下是测试人员最有用的方法。

方法 详情
detectChanges 为组件触发变更检测周期。
调用它以初始化组件（它会调用 ngOnInit），并在测试代码之后，更改组件的数据绑定属性值。Angular 看不到您已更改 personComponent.name，并且在您调用 detectChanges 之前不会更新 name 绑定。
之后运行 checkNoChanges 以确认没有循环更新，除非以 detectChanges(false) 形式调用；
autoDetectChanges 当您希望 fixture 自动检测变更时，请将其设置为 true。
当 autodetect 为 true 时，测试 fixture 会在创建组件后立即调用 detectChanges。然后，它会侦听相关的 Zone 事件并相应地调用 detectChanges。当您的测试代码直接修改组件属性值时，您可能仍然需要调用 fixture.detectChanges 以触发数据绑定更新。
默认值为 false。喜欢精细控制测试行为的测试人员倾向于将其保持为 false。
checkNoChanges 执行变更检测运行，以确保没有待处理的变更。如果有，则抛出异常。
isStable 如果此测试夹具当前稳定，则返回 true。如果存在尚未完成的异步任务，则返回 false。
whenStable 返回一个 Promise，该 Promise 在测试夹具稳定时求解。
要在异步活动或异步变更检测完成后恢复测试，请钩住该 Promise。请参阅 whenStable。
destroy 触发组件销毁。

DebugElement

DebugElement 提供了对组件 DOM 表示的关键洞察。

从由 fixture.debugElement 返回的测试根组件的 DebugElement，你可以遍历（并查询）fixture 的整个元素和组件子树。

以下是对于测试人员最有用的 DebugElement 成员，按大致实用性顺序排列：

成员 详情
nativeElement 浏览器中对应的 DOM 元素
query 调用 query(predicate: Predicate<DebugElement>) 会返回子树中任何深度下与 谓词匹配的第一个 DebugElement。
queryAll 调用 queryAll(predicate: Predicate<DebugElement>) 会返回所有与子树中任何深度的 谓词 匹配的 DebugElement。
injector 宿主依赖注入器。例如，根元素的组件实例注入器。
componentInstance 元素自身的组件实例（如果有）。
context 一个为该元素提供父级上下文的对象。通常是管理该元素的祖先组件实例。
当一个元素在 @for 块中重复时，上下文是 RepeaterContext，其 $implicit 属性是行实例值的该行实例值。例如，@for(hero of heroes; ...) 中的 hero。
children 直接的 DebugElement 子项。通过向下遍历 children 来遍历树。 DebugElement 还有 childNodes，这是一个 DebugNode 对象列表。 DebugElement 源自 DebugNode 对象，并且节点数量通常多于元素。测试人员通常可以忽略普通节点。
parent DebugElement 父项。如果是根元素，则为 null。
name 元素标签名称（如果它是元素）。
triggerEventHandler 如果元素的 listeners 集合中存在相应的侦听器，则按名称触发事件。第二个参数是处理程序期望的事件对象。请参阅 triggerEventHandler。
如果事件缺少侦听器或存在其他问题，请考虑调用 nativeElement.dispatchEvent(eventObject)。
listeners 附加到组件的 @Output 属性和/或元素的事件属性的回调。
providerTokens 此组件的注入器查找令牌。包括组件本身以及组件在其 providers 元数据中列出的令牌。
source 在源组件模板中查找此元素的位置。
references 与模板局部变量（例如，#foo）关联的对象字典，按局部变量名键控。

DebugElement.query(predicate) 和 DebugElement.queryAll(predicate) 方法接受一个谓词，该谓词会过滤源元素的子树以匹配 DebugElement。

Angular 的 By 类有三个用于常见谓词的静态方法：

静态方法 详情
By.all 返回所有元素
By.css(selector) 返回具有匹配 CSS 选择器的元素
By.directive(directive) 返回 Angular 匹配到指令类实例的元素
