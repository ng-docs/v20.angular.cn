组件测试基础

edit

与 Angular 应用的所有其他部分不同，组件将 HTML 模板和 TypeScript 类结合在一起。 组件实际上是模板和类协同工作的产物。 为了充分地测试组件，你应该测试它们是否按预期协同工作。

此类测试需要像 Angular 那样在浏览器 DOM 中创建组件的宿主元素，并根据组件模板的描述，研究组件类与 DOM 之间的交互。

Angular 的 TestBed 促进了此类测试，你将在后续章节中看到。 但在许多情况下，单独测试组件类，而不涉及 DOM，可以以一种直接、更明显的方式验证组件的大部分行为。

组件 DOM 测试

组件不仅仅是它的类。 组件还会与 DOM 和其他组件交互。 仅凭类本身无法告诉你组件是否会正确渲染、响应用户输入和手势，或者与其父组件和子组件集成。

  • Lightswitch.clicked() 是否绑定到任何东西，以便用户可以调用它？
  • Lightswitch.message 是否显示？
  • 用户是否真的可以选择 DashboardHeroComponent 显示的英雄？
  • 英雄的名字是否按预期显示（例如大写）？
  • WelcomeComponent 模板中的欢迎消息是否显示？

对于前面阐明的简单组件来说，这些可能不是令人困扰的问题。 但是，许多组件与其模板中描述的 DOM 元素具有复杂的交互，导致 HTML 随着组件状态的变化而出现和消失。

要回答这类问题，你必须创建与组件关联的 DOM 元素，必须检查 DOM 以确认组件状态在适当的时候正确显示，并且必须模拟用户与屏幕的交互，以确定这些交互是否会导致组件按预期运行。

要编写这类测试，你将使用 TestBed 的附加功能以及其他测试辅助工具。

CLI 生成的测试

默认情况下，当你要求 CLI 生成新组件时，它会为你创建一个初始测试文件。

例如，以下 CLI 命令在 app/banner 文件夹中生成一个 BannerComponent（带有内联模板和样式）：

ng generate component banner --inline-template --inline-style --module app

它还会为组件生成一个初始测试文件 banner-external.component.spec.ts，如下所示：

减少建立过程

此文件中只有最后三行实际测试了组件，它们所做的只是断言 Angular 可以创建该组件。

TestBed.createComponent() 创建 BannerComponent 的实例，将相应的元素添加到测试运行器 DOM 中，并返回一个 ComponentFixture。

重要：调用 createComponent 后，请勿重新配置 TestBed。

createComponent 方法会冻结当前的 TestBed 定义，使其无法进行进一步配置。

nativeElement

ComponentFixture.nativeElement 的值类型为 any。 稍后你会遇到 DebugElement.nativeElement，它的类型也是 any。

Angular 在编译时无法知道 nativeElement 是哪种 HTML 元素，甚至无法知道它是否是 HTML 元素。 该应用可能正在非浏览器平台（例如服务器或 Web Worker）上运行，在这些平台上，该元素可能具有缩减的 API，或者根本不存在。

本指南中的测试旨在在浏览器中运行，因此 nativeElement 值将始终是 HTMLElement 或其派生类之一。

知道它某种类型的 HTMLElement，请使用标准 HTML querySelector 更深入地探查元素树。

这种迂回的元素路径是有充分理由的。

nativeElement 的属性取决于运行时环境。 你可能正在非浏览器平台上运行这些测试，该平台没有 DOM 或其 DOM 模拟不支持完整的 HTMLElement API。

Angular 依赖 DebugElement 抽象，以在所有支持的平台上安全地工作。 Angular 不创建 HTML 元素树，而是创建 DebugElement 树，该树包装了运行时平台的原生元素。 nativeElement 属性会解开 DebugElement 并返回特定于平台的元素对象。

由于本指南的示例测试旨在仅在浏览器中运行，因此这些测试中的 nativeElement 始终是一个 HTMLElement，你可以在测试中探索其熟悉的方法和属性。

By.css()

尽管本指南中的测试都在浏览器中运行，但有些应用可能至少在某些时候会在不同的平台上运行。

例如，作为一种让应用在网络连接不良的设备上启动更快的策略，组件可能首先在服务器上渲染。服务器端渲染器可能不支持完整的 HTML 元素 API。如果它不支持 querySelector，则之前的测试可能会失败。

DebugElement 提供了适用于所有支持平台的查询方法。 这些查询方法接受一个谓词函数，当 DebugElement 树中的节点匹配选择标准时，该函数返回 true。

一些值得注意的观察结果：

  • By.css() 静态方法使用 标准 CSS 选择器选择 DebugElement 节点。
  • 查询返回段落的 DebugElement。
  • 你必须解包该结果才能获得段落元素。

当你通过 CSS 选择器进行过滤，并且只测试浏览器原生元素的属性时，By.css 方法可能有点大材小用。

使用标准的 HTMLElement 方法（如 querySelector() 或 querySelectorAll()）进行过滤通常更直接和清晰。
