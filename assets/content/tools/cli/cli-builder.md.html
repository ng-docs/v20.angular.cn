Angular CLI 构建器

edit

许多 Angular CLI 命令会在你的代码上运行一个复杂的过程，例如构建、测试或为你的应用启动服务器。 这些命令使用一个名为 Architect 的内部工具来运行 CLI 构建器，这些构建器会调用另一个工具（打捆器、测试运行器、服务器）来完成期望的任务。 自定义构建器可以执行全新的任务，或者更改现有命令使用的第三方工具。

本文档介绍了 CLI 构建器如何与工作区配置文件集成，并展示了如何创建你自己的构建器。

提示： 在这个 GitHub 代码仓中查找此处使用的范例代码。

CLI 构建器

内部的 Architect 工具将工作委派给名为构建器的处理函数。 构建器处理函数接收两个参数：

参数 类型
options JSONObject
context BuilderContext

这里关注点分离与原理图相同，原理图用于处理其他会触及你代码的 CLI 命令（例如 ng generate）。

  • options 对象由 CLI 用户的选项和配置提供，而 context 对象由 CLI Builder API 自动提供。
  • 除了上下文信息之外，context 对象还提供对调度方法 context.scheduleTarget() 的访问。 调度器使用给定的目标配置执行构建器处理函数。

构建器处理函数可以是同步的（返回值）、异步的（返回一个 Promise），或者观察并返回多个值（返回一个 Observable）。 返回值必须始终是 BuilderOutput 类型。 此对象包含一个布尔值 success 字段和一个可选的 error 字段，其中可以包含错误消息。

Angular 提供了一些构建器，CLI 将它们用于 ng build 和 ng test 等命令。 这些构建器以及其他内置 CLI 构建器的默认目标配置可以在工作区配置文件 angular.json 的 "architect" 部分中找到和配置。 此外，你还可以通过创建自己的构建器来扩展和自定义 Angular，你可以使用 ng run CLI 命令直接运行它们。

构建器项目结构

构建器位于一个 "project" 文件夹中，该文件夹的结构类似于 Angular 工作区，全局配置文件位于顶层，更具体的配置位于包含定义行为的代码文件的 source 文件夹中。 例如，你的 myBuilder 文件夹可能包含以下文件。

文件 用途
src/my-builder.ts 构建器定义的主源码文件。
src/my-builder.spec.ts 测试的源码文件。
src/schema.json 构建器输入选项的定义。
builders.json 构建器定义。
package.json 依赖项。请参阅 https://docs.npmjs.com/files/package.json。
tsconfig.json TypeScript 配置。

构建器可以发布到 npm，请参阅发布你的库。

创建构建器

例如，创建一个将文件复制到新位置的构建器。 要创建构建器，请使用 createBuilder() CLI Builder 函数，并返回一个 Promise<BuilderOutput> 对象。

src/my-builder.ts (构建器骨架)

import {BuilderContext, BuilderOutput, createBuilder} from '@angular-devkit/architect';import {JsonObject} from '@angular-devkit/core';import {promises as fs} from 'fs';interface Options extends JsonObject {  source: string;  destination: string;}export default createBuilder(copyFileBuilder);async function copyFileBuilder(options: Options, context: BuilderContext): Promise<BuilderOutput> {  context.reportStatus(`Copying ${options.source} to ${options.destination}.`);  try {    await fs.copyFile(options.source, options.destination);  } catch (err) {    context.logger.error('Failed to copy file.');    return {      success: false,      error: (err as Error).message,    };  }  context.reportStatus('Done.');  return {success: true};}

现在让我们为其添加一些逻辑。 以下代码从用户选项中检索源文件路径和目标文件路径，并将文件从源复制到目标（使用内置 NodeJS copyFile() 函数的 Promise 版本）。 如果复制操作失败，它将返回一个错误，其中包含有关底层问题的消息。

src/my-builder.ts (构建器)

import {BuilderContext, BuilderOutput, createBuilder} from '@angular-devkit/architect';import {JsonObject} from '@angular-devkit/core';import {promises as fs} from 'fs';interface Options extends JsonObject {  source: string;  destination: string;}export default createBuilder(copyFileBuilder);async function copyFileBuilder(options: Options, context: BuilderContext): Promise<BuilderOutput> {  context.reportStatus(`Copying ${options.source} to ${options.destination}.`);  try {    await fs.copyFile(options.source, options.destination);  } catch (err) {    context.logger.error('Failed to copy file.');    return {      success: false,      error: (err as Error).message,    };  }  context.reportStatus('Done.');  return {success: true};}

处理输出

默认情况下，copyFile() 不会向进程标准输出或错误输出任何内容。 如果发生错误，可能很难理解当问题发生时构建器到底在尝试做什么。 通过使用 Logger API 记录其他信息来添加一些额外的上下文。 即使标准输出和错误被禁用，这也允许构建器本身在单独的进程中执行。

你可以从上下文中检索 Logger 实例。

进度和状态报告

CLI Builder API 包括进度和状态报告工具，这些工具可以为某些函数和接口提供提示。

要报告进度，请使用 context.reportProgress() 方法，该方法接受当前值、可选的总数和状态字符串作为参数。 总数可以是任何数字。例如，如果你知道必须处理多少个文件，则总数可以是文件数，而当前值应该是到目前为止已处理的文件数。 除非你传入新的字符串值，否则状态字符串不会被修改。

在我们的范例中，复制操作要么完成，要么仍在执行，因此不需要进度报告，但是你可以报告状态，以便调用我们构建器的父构建器知道发生了什么。 使用 context.reportStatus() 方法生成任意长度的状态字符串。

提示： 不能保证长字符串会完整显示；它可能会被截断以适应显示它的 UI。

传入一个空字符串以删除状态。

src/my-builder.ts (进度报告)

import {BuilderContext, BuilderOutput, createBuilder} from '@angular-devkit/architect';import {JsonObject} from '@angular-devkit/core';import {promises as fs} from 'fs';interface Options extends JsonObject {  source: string;  destination: string;}export default createBuilder(copyFileBuilder);async function copyFileBuilder(options: Options, context: BuilderContext): Promise<BuilderOutput> {  context.reportStatus(`Copying ${options.source} to ${options.destination}.`);  try {    await fs.copyFile(options.source, options.destination);  } catch (err) {    context.logger.error('Failed to copy file.');    return {      success: false,      error: (err as Error).message,    };  }  context.reportStatus('Done.');  return {success: true};}

构建器输入

你可以通过 CLI 命令（例如 ng build）间接调用构建器，也可以使用 Angular CLI ng run 命令直接调用。 在任何一种情况下，你都必须提供必要的输入，但可以让其他输入默认为为特定目标预先配置的值，该目标由配置指定，或在命令行上设置。

输入验证

你在与该构建器关联的 JSON 模式中定义构建器输入。 与原理图类似，Architect 工具将求解后的输入值收集到 options 对象中，并在将它们传递给构建器函数之前，根据模式验证它们的类型。

对于我们的范例构建器，options 应该是一个具有两个键的 JsonObject：source 和 destination，每个键都是一个字符串。

你可以提供以下模式来验证这些值的类型。

src/schema.json

{  "$schema": "http://json-schema.org/schema",  "type": "object",  "properties": {    "source": {      "type": "string"    },    "destination": {      "type": "string"    }  }}

提示： 这是一个最小化的范例，但是使用模式进行验证可能非常强大。 有关更多信息，请参阅 JSON 模式网站。

要将我们的构建器实现与其模式和名称链接起来，你需要创建一个构建器定义文件，你可以在 package.json 中指向它。

创建一个名为 builders.json 的文件，如下所示：

builders.json

{  "builders": {    "copy": {      "implementation": "./dist/my-builder.js",      "schema": "./src/schema.json",      "description": "Copies a file."    }  }}

在 package.json 文件中，添加一个 builders 键，告诉 Architect 工具在哪里可以找到我们的构建器定义文件。

package.json

{  "name": "@example/copy-file",  "version": "1.0.0",  "description": "Builder for copying files",  "builders": "builders.json",  "dependencies": {    "@angular-devkit/architect": "~0.1200.0",    "@angular-devkit/core": "^12.0.0"  }}

我们构建器的正式名称现在是 @example/copy-file:copy。 第一部分是包名，第二部分是在 builders.json 文件中指定的构建器名称。

这些值在 options.source 和 options.destination 上访问。

src/my-builder.ts (报告状态)

目标配置

构建器必须具有已定义的目标，该目标将其与特定的输入配置和项目关联。

目标在 angular.json CLI 配置文件中定义。 一个目标指定了要使用的构建器、它的默认选项配置以及命名的备用配置。 Angular CLI 中的 Architect 使用目标定义来求解给定运行的输入选项。

angular.json 文件为每个项目都有一个 section，并且每个项目的 "architect" section 为构建器配置目标，这些构建器被诸如 'build'、'test' 和 'serve' 之类的 CLI 命令所使用。 例如，默认情况下，ng build 命令运行构建器 @angular-devkit/build-angular:browser 来执行构建任务，并传入为 angular.json 中 build 目标指定的默认选项值。

angular.json

…"myApp": {  …  "architect": {    "build": {      "builder": "@angular-devkit/build-angular:browser",      "options": {        "outputPath": "dist/myApp",        "index": "src/index.html",        …      },      "configurations": {        "production": {          "fileReplacements": [            {              "replace": "src/environments/environment.ts",              "with": "src/environments/environment.prod.ts"            }          ],          "optimization": true,          "outputHashing": "all",          …        }      }    },    …  }}…

该命令将 "options" section 中指定的一组默认选项传递给构建器。 如果你传递 --configuration=production 标志，它将使用 production 配置中指定的重写值。 在命令行中单独指定进一步的选项重写。

目标字符串

通用的 ng run CLI 命令将其第一个参数作为以下形式的目标字符串。

project:target[:configuration]
详情
project 目标与之关联的 Angular CLI 项目的名称。
target 来自 angular.json 文件的 architect section 的命名构建器配置。
configuration （可选）给定目标的特定配置重写的名称，如 angular.json 文件中所定义。

如果你的构建器调用了另一个构建器，它可能需要读取传递的目标字符串。 通过使用 @angular-devkit/architect 中的 targetFromTargetString() 实用函数将此字符串解析为对象。

调度和运行

Architect 异步运行构建器。 要调用构建器，你需要调度一个任务，使其在所有配置求解完成后运行。

在调度器返回 BuilderRun 控制对象之前，构建器函数不会执行。 CLI 通常通过调用 context.scheduleTarget() 函数来调度任务，然后使用 angular.json 文件中的目标定义来求解输入选项。

Architect 通过获取默认选项对象来求解给定目标的输入选项，然后从配置中覆盖值，然后再从传递给 context.scheduleTarget() 的 overrides 对象中进一步覆盖值。 对于 Angular CLI，overrides 对象是从命令行参数构建的。

Architect 根据构建器的模式验证生成的选项值。 如果输入有效，Architect 将创建上下文并执行构建器。

更多信息请参见 工作区配置。

提示： 你也可以通过调用 context.scheduleBuilder() 从另一个构建器或测试中直接调用构建器。 你可以将 options 对象直接传递给该方法，并且这些选项值将根据构建器的模式进行验证，而无需进一步调整。

只有 context.scheduleTarget() 方法通过 angular.json 文件求解配置和 overrides。

添加目标

添加一个新目标，它将运行我们的构建器来复制文件。 此目标告诉构建器复制 package.json 文件。

  • 我们将为项目的 architect 对象添加一个新的目标 section
  • 名为 copy-package 的目标使用我们的构建器，你已将其发布到 @example/copy-file。
  • options 对象为定义的两个输入提供默认值。
    • source - 您正在复制的现有文件。
    • destination - 您要复制到的路径。

angular.json

{  "projects": {    "builder-test": {      "architect": {        "copy-package": {          "builder": "@example/copy-file:copy",          "options": {            "source": "package.json",            "destination": "package-copy.json"          }        },        // Existing targets...      }    }  }}

运行构建器

要使用新目标的默认配置运行我们的构建器，请使用以下 CLI 命令。

ng run builder-test:copy-package

这会将 package.json 文件复制到 package-copy.json。

使用命令行参数来重写配置的默认值。 例如，要使用不同的 destination 值运行，请使用以下 CLI 命令。

ng run builder-test:copy-package --destination=package-other.json

这会将文件复制到 package-other.json 而不是 package-copy.json。 因为你没有重写 source 选项，它仍将从默认的 package.json 文件复制。

测试构建器

为你的构建器使用集成测试，以便你可以使用 Architect 调度器来创建上下文，如这个范例中所示。 在构建器源码目录中，创建一个新的测试文件 my-builder.spec.ts。该测试创建了 JsonSchemaRegistry（用于模式验证）、TestingArchitectHost（ArchitectHost 的内存实现）和 Architect 的新实例。

这是一个运行复制文件构建器的测试范例。 该测试使用构建器复制 package.json 文件，并验证复制文件的内容与源码是否相同。

src/my-builder.spec.ts

import {Architect} from '@angular-devkit/architect';import {TestingArchitectHost} from '@angular-devkit/architect/testing';import {schema} from '@angular-devkit/core';import {promises as fs} from 'fs';import {join} from 'path';describe('Copy File Builder', () => {  let architect: Architect;  let architectHost: TestingArchitectHost;  beforeEach(async () => {    const registry = new schema.CoreSchemaRegistry();    registry.addPostTransform(schema.transforms.addUndefinedDefaults);    // TestingArchitectHost() takes workspace and current directories.    // Since we don't use those, both are the same in this case.    architectHost = new TestingArchitectHost(__dirname, __dirname);    architect = new Architect(architectHost, registry);    // This will either take a Node package name, or a path to the directory    // for the package.json file.    await architectHost.addBuilderFromPackage(join(__dirname, '..'));  });  it('can copy files', async () => {    // A "run" can have multiple outputs, and contains progress information.    const run = await architect.scheduleBuilder('@example/copy-file:copy', {      source: 'package.json',      destination: 'package-copy.json',    });    // The "result" member (of type BuilderOutput) is the next output.    const output = await run.result;    // Stop the builder from running. This stops Architect from keeping    // the builder-associated states in memory, since builders keep waiting    // to be scheduled.    await run.stop();    // Expect that the copied file is the same as its source.    const sourceContent = await fs.readFile('package.json', 'utf8');    const destinationContent = await fs.readFile('package-copy.json', 'utf8');    expect(destinationContent).toBe(sourceContent);  });});

提示： 在你的 repo 中运行此测试时，你需要 ts-node 包。 你可以通过将 my-builder.spec.ts 重命名为 my-builder.spec.js 来避免这种情况。

监听模式

大多数构建器运行一次并返回。但是，这种行为与监听更改的构建器（例如 devserver）并不完全兼容。 Architect 可以支持监听模式，但有一些事项需要注意。

  • 要与监听模式一起使用，构建器处理函数应返回一个 Observable。 Architect 订阅该 Observable 直到它完成，如果使用相同的参数再次调度构建器，则可能会复用它。

  • 构建器应始终在每次执行后发出一个 BuilderOutput 对象。 一旦执行完毕，它就可以进入监听模式，以由外部事件触发。 如果某个事件触发它重新启动，则构建器应执行 context.reportRunning() 函数，以告知 Architect 它正在再次运行。 这可以防止 Architect 在调度另一个运行时停止构建器。

当你的构建器调用 BuilderRun.stop() 以退出监听模式时，Architect 会取消订阅构建器的 Observable 并调用构建器的 teardown 逻辑进行清理。 此行为还允许停止和清理长时间运行的构建。

通常，如果你的构建器正在监听外部事件，则应将运行分为三个阶段。

阶段 详情
正在运行 正在执行的任务，例如调用编译器。当编译器完成并且你的构建器发出 BuilderOutput 对象时，此阶段结束。
正在监视 在两次运行之间，监视外部事件流。例如，监视文件系统中的任何更改。当编译器重新启动并且调用 context.reportRunning() 时，此过程结束。
完成 任务已完全完成（例如，编译器需要运行多次），或者构建器运行已停止（使用 BuilderRun.stop()）。Architect 会执行拆卸逻辑并取消订阅你的构建器的 Observable。

总结

CLI 构建器 API 提供了一种通过使用构建器执行自定义逻辑来更改 Angular CLI 行为的方法。

  • 构建器可以是同步的或异步的，可以执行一次或监视外部事件，并且可以调度其他构建器或目标。
  • 构建器在 angular.json 配置文件中指定了选项默认值，这些默认值可以被目标的备用配置覆盖，并可以进一步被命令行标志覆盖
  • Angular 团队建议你使用集成测试来测试 Architect 构建器。使用单元测试来验证构建器执行的逻辑。
  • 如果你的构建器返回一个 Observable，它应该在该 Observable 的拆卸逻辑中清理构建器。
